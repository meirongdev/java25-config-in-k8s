# GC 对比实验报告：G1GC vs ZGC

**实验日期：** 2026-03-11
**环境：** Java 25 + Spring Boot 3.5.1 + kind 3节点集群
**负载：** k6 × 10 VUs × 60s，60% 内存压力（30MB 短生命周期对象）+ 40% CPU 压力（0.3s 质数计算）
**JVM 参数（公共）：** `-XX:MaxRAMPercentage=75`

## 二、关键发现

### 发现 1：ZGC 的 GC 停顿几乎为零，但平均吞吐始终低于 G1GC

| 资源组 | G1GC GC 总耗时 | ZGC GC 总耗时 | G1GC 平均延迟 | ZGC 平均延迟 |
|--------|--------------|-------------|------------|------------|
| 1c/1Gi | 1,574 ms | 5–7 ms | 7.6 ms | 9.5–9.7 ms |
| 2c/2Gi | 1,097 ms | 0 ms | 7.9 ms | 9.2–9.6 ms |
| 4c/4Gi | 668 ms | 0–1 ms | 7.5 ms | 10.1–10.3 ms |

**ZGC 的代价是固定的：约 +2 ms 平均延迟，-2 RPS。**
**这个代价不随 CPU 增加而减小**——即使到 4c，ZGC 的平均延迟依然比 G1GC 高 ~2.5 ms。

原因：ZGC 的并发 GC 线程和 Load Barrier（读屏障）始终消耗一定量的 CPU 和内存带宽，这是其算法设计的固有成本，与资源总量关系不大。

### 发现 2：G1GC 的 p99 随 CPU 增加显著改善，ZGC 则不然

| 资源组 | G1GC p99 | ZGC p99 |
|--------|---------|---------|
| 1c/1Gi | 121 ms | 243 ms |
| 2c/2Gi | 93 ms | 128 ms |
| 4c/4Gi | **87 ms** | 242 ms |

- G1GC p99：CPU 增加后从 121ms 降至 87ms，**改善 28%**，趋势稳定
- ZGC p99：4c 时反而比 2c 更高（242ms），说明在我们的短生命周期对象负载下，ZGC 的偶发大回收事件受随机性影响较大

## 三、结论与选型建议

### 何时选 G1GC

- **标准微服务，追求平均低延迟和高吞吐**
- CPU < 2c 的受限环境（ZGC 并发线程代价相对更大）
- 对 p99 有严格要求但允许偶发短停顿的场景（交易链路非核心路径）

**配置：** `-XX:+UseG1GC -XX:MaxRAMPercentage=75`

### 何时选 ZGC

- **对最大停顿（Max GC Pause）有严格要求**（如 SLA ≤ 5ms）
- 大堆应用（> 4GB）或长生命周期对象较多（老年代压力大）
- 可以接受平均延迟略高 ~2ms 的场景

**配置：** `-XX:+UseZGC -XX:MaxRAMPercentage=75`

### 2026 年 Java 25 选型矩阵

| 场景 | GC | 理由 |
|------|----|----|
| CPU < 1c | G1GC | ZGC 并发线程在低 CPU 下代价更大 |
| 标准微服务（1-2c） | G1GC | 平均延迟更低，p99 更稳定 |
| 低延迟核心服务（SLA < 5ms max pause） | ZGC | GC max pause 0-1ms |
| 大内存应用（> 8GB heap） | ZGC | G1GC 在大堆下停顿更长 |
| 批处理 | ParallelGC | 吞吐优先，停顿不敏感 |

---

## 四、注意事项

1. **本实验负载为短生命周期对象**——最有利于 G1GC 的场景（Young GC 快速回收）。在有大量长生命周期对象或大堆的生产环境中，ZGC 的优势会更显著。

2. **ZGC 的堆 jvm.memory.max 数值偏高**（约 2x），不代表实际内存消耗，是虚拟地址映射的统计差异。

3. **种集群（kind）在本地 macOS 的 CPU 隔离精度低于生产 Kubernetes**，实际生产环境中 CPU 节流效果可能更明显，ZGC 的 GC 停顿优势会更突出。
